标准工作流程

当你在你的项目上工作时，你会自然而然地创建精灵，编辑脚本，删除时间线，以及其他任何随着项目发展而需要的东西。这些都是你可能想要跟踪的行动，并且在出现问题时能够得到反馈。 出错。这是使用源码控制的主要原因--这样任何意外都可以被还原--我们将迅速带你从头开始了解基本的工作流程，这样你就会明白这是如何工作的。

从头开始，创建一个新的GML项目，然后通过主选项激活源控制。这样我们就可以进入顶部的源控制菜单，所以我们马上去 选择创建项目库。如果你看一下房间资源，你可以看到默认的房间已经获得了一个红色的感叹号 ，这意味着它已经被修改了。 意味着它已经从上次的状态被修改了 。如果我们在项目中还有其他资源，你会看到它们都有同样的图标，如下图所示。

我们现在可以选择提交空白项目，或者把它变成一个基本状态。现在，我们要做的是第一次 提交，所以点击Source Control>Commit Changes。现在我们会看到一个新的窗口，它被分成三个区域。

  • 阶段性变化。这代表将被提交到源代码的变化。这可能是空白的，但它可能会显示一个主选项的变化，这很好。
  • 未归档的变更。这显示了已经改变的文件，但我们还没有告诉 Git 我们要提交它们 - 它们是 "待定 "的改变。
  • 提交信息。这是我们可以添加的说明，以解释什么是变化。

现在，点击Stage All按钮，因为我们想提交所有的修改，然后在Commit Message中输入类似 "First Commit!"的字样，并点击Commit。现在我们所有的资源都应该有一个绿色的勾 ，意味着没有发现任何修改。

值得注意的是，当你在这个窗口中工作时，各种暂存 和未暂存的 信息将以一个符号为前缀。所使用的符号将根据操作和文件的不同而不同，以下是所使用的符号 正在使用。

  • [...] - 阶段性变化（以前是[A]）。
  • [+] - 新添加
  • [->] - 重新命名/移动
  • [-] - 已删除
  • [M] - 非阶段性变化
  • [?] - 未被追踪的文件
  • [!] - 缺少文件

我们的工作进程现在可以开始了，我们可以开始建立我们的项目，并保存和提交修改，就像我们在上面几段所做的那样。
